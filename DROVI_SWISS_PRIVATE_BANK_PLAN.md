# Drovi — Swiss Private Bank of Truth (Master Plan)

## 1) Strategic Positioning
Drovi becomes an **institutional trust infrastructure**, not an AI productivity app.

Core promise:
- Nothing material is forgotten.
- Nothing material is asserted without evidence.
- Nothing material changes without trace.

Positioning line:
- **Drovi is a fiduciary memory system for institutions.**

Emotional outcome for users:
- Calm
- Control
- Authority
- Auditability

## 2) Brand Doctrine (Non-Negotiables)
1. Restraint over novelty.
2. Evidence over prose.
3. Permanence over speed theater.
4. Precision over possibility language.
5. Discretion over visibility.
6. Ritual over growth hacks.

This doctrine must shape product, infra, support, and founder behavior.

## 3) Product Reframe: From App to Institutional Vault
Drovi should be experienced as a vault with controlled windows into truth.

### 3.1 Core surfaces (renamed)
- `Reality Stream` → **Record Feed**
- `Tasks` → **Commitment Register**
- `Decisions` → **Decision Ledger**
- `Claims` → **Assertion Register**
- `Trust & Audit` → **Integrity Office**
- `Connected Sources` → **Custody Connectors**
- `Exchange` → **Mandate Exchange**

### 3.2 Proof-first output contract
Every object shown to users must include:
- Origin source(s)
- Evidence snippets and hash IDs
- Extraction timestamp
- Confidence and calibration reason
- Supersession status
- Responsible actor or unresolved identity state

### 3.3 New premium primitives
- **Record Certificate**: signed export proving provenance + integrity chain for any timeline, client, matter, or incident.
- **Continuity Score**: institutional continuity KPI (coverage, contradiction rate, unresolved commitments, stale mandates).
- **Dispute Mode**: compare two versions of truth across time and generate evidence-backed variance report.
- **Custodian Notes**: immutable operator notes attached to high-risk records.

## 4) Frontend Experience Architecture (Old Money System)
Use the `frontend-design` skill principles but tuned for institutional restraint.

### 4.1 Design language
- Palette: deep forest, midnight navy, parchment, stone, antique gold accents.
- Typography: high-credibility serif headlines + refined sans body.
- Motion: near-instant, subtle fades and position shifts only.
- Visual weight: editorial spacing, calm density, low-noise surfaces.
- No gradients, no neon, no playful illustrations, no dashboard chaos.

### 4.2 UX behavior standards
- Every view must answer:
  - What changed?
  - Why it matters?
  - What evidence supports it?
  - What action is now required?
- Empty/error states become formal operational states:
  - “No admissible evidence found.”
  - “Source under review.”
  - “Record pending verification.”

### 4.3 Interaction models
- **Command Bar** becomes **Mandate Bar** with institutional verbs:
  - “Open decision trail for matter X”
  - “Show superseded commitments from last quarter”
  - “Prepare briefing for partner review”
- Side panels become dossier sheets with fixed sections:
  - Summary
  - Evidence
  - Timeline
  - Supersession
  - Access history

### 4.4 Vertical skin system
Create theme + vocabulary overlays per vertical:
- Legal: Matters, Advice, Privilege, Contradiction Radar
- Accounting: Engagements, Filing Obligations, Missing Inputs, Deadline Drift
- Government: Policy Dossiers, Approval Chains, Public Record Trails
- Construction: Change Orders, RFIs, Site Commitments, Contractor Risk

## 5) Backend Trust Architecture (Swiss-grade)

### 5.1 Chain-of-custody by default
- Every ingest event gets a content hash and custody metadata.
- Build Merkle-chain snapshots per tenant/day.
- Optionally notarize root hash to external timestamp authority.

### 5.2 Crypto posture
- Per-tenant encryption domains.
- BYOK/HYOK option for enterprise tiers.
- Envelope encryption for evidence objects and sensitive metadata.
- Key rotation with non-destructive rewrap workflows.

### 5.3 Identity and access hardening
- SSO-first policy (SAML/OIDC), password fallback only for sandbox.
- RBAC + ABAC for object, evidence, and field-level access.
- Just-in-time privilege for high-risk actions.
- Admin actions require reason codes and are always auditable.

### 5.4 Policy engine and governance
- Central policy decision point for:
  - Evidence visibility
  - Export permissions
  - Cross-border access rules
  - Redaction requirements
- “No evidence, no persist” remains hard guardrail for high-stakes entities.

### 5.5 Deployment trust tiers
- SaaS shared control plane
- Dedicated single-tenant cloud
- Fully private/on-prem package (air-gapped capable)

## 6) Data Integrity and Reliability Contracts

### 6.1 SLO baseline (institutional grade)
- Read p95: < 200ms on key pages
- Ask first token p95: < 250ms
- Ingestion lag p95: < 60s for live channels
- Pipeline durability: no silent drops, deterministic replay

### 6.2 Reliability controls
- Canonical event log (Kafka) with replay and DLQ workflows.
- Durable job state with leases, retries, and operator overrides.
- Evidence ingest idempotency by source IDs + content hash.
- Mandatory backpressure controls and sync budget per connector.

### 6.3 Operability
- Gold dashboards per tenant:
  - Source freshness
  - Contradiction delta
  - Coverage of commitments/decisions/tasks
  - Evidence retrieval latency
- Red/amber/green trust status at org and source levels.

## 7) Security as Client Experience
Security must be visible as calm confidence, not fear messaging.

Client-facing capabilities:
- Access provenance logs
- Data residency options
- Immutable audit exports
- Per-tenant retention/deletion policy controls
- Legal hold mode
- Signed “Integrity Statement” generated on demand

## 8) Service Model and Rituals (White-Glove)

### 8.1 Onboarding ritual
- Private briefing request (no generic free trial for target tiers).
- Structured sequence:
  - Security review
  - Data custody mapping
  - Pilot mandate and success criteria
  - Controlled launch window

### 8.2 Operating cadence per client
- Weekly operations brief
- Monthly integrity report
- Quarterly continuity review

### 8.3 Escalation protocol
- Named human custodian per pilot/enterprise account.
- Incident response with evidence-backed post-mortem dossier.

## 9) Language and Tone System
Ban noisy startup language in product and sales copy.

Preferred lexicon:
- Record
- Trail
- Custody
- Integrity
- Supersession
- Continuity
- Mandate
- Evidence
- Register
- Dossier

Writing rules:
- Declarative statements.
- No hype adjectives.
- No “AI thinks”.
- Always tie statements to evidence or policy.
